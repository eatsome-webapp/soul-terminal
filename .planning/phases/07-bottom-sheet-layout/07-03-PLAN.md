<task id="07-03-01">
<title>Verify and fix back button callback priority with DrawerLayout</title>
<read_first>
- app/src/main/java/com/termux/app/TermuxActivity.java (setupBottomSheet method — the OnBackPressedCallback added in 07-01-02)
- .planning/phases/07-bottom-sheet-layout/07-RESEARCH.md (§6 Back Button Migration)
</read_first>
<action>
The OnBackPressedCallback was added in Plan 07-01 inside `setupBottomSheet()`. Verify the callback handles these scenarios correctly:

1. **Drawer open + sheet expanded:** Back should close drawer first, then next back closes sheet.
2. **Sheet expanded, drawer closed:** Back should go to STATE_COLLAPSED.
3. **Sheet half-expanded, drawer closed:** Back should go to STATE_COLLAPSED.
4. **Sheet collapsed, drawer closed:** Back should close the app (finishActivityIfNotFinishing).
5. **Sheet hidden:** Back should close the app.

Review the callback logic in `setupBottomSheet()`. The current implementation from 07-01-02 checks drawer first, which is correct. Ensure the following edge case is handled:

If the DrawerLayout `getDrawer()` returns null early in lifecycle (before layout inflation completes), the callback would NPE. Add a null-safety check:

```java
DrawerLayout drawer = getDrawer();
if (drawer != null && drawer.isDrawerOpen(Gravity.LEFT)) {
    drawer.closeDrawers();
    return;
}
```

Also verify: when the sheet is in `STATE_DRAGGING` or `STATE_SETTLING` (transient states), back should still work. The callback should check for these states and treat them as "not collapsed":

Update the back callback logic to:
```java
@SuppressLint("RtlHardcoded")
@Override
public void handleOnBackPressed() {
    DrawerLayout drawer = getDrawer();
    if (drawer != null && drawer.isDrawerOpen(Gravity.LEFT)) {
        drawer.closeDrawers();
        return;
    }
    int state = mBottomSheetBehavior.getState();
    if (state == BottomSheetBehavior.STATE_EXPANDED
            || state == BottomSheetBehavior.STATE_HALF_EXPANDED
            || state == BottomSheetBehavior.STATE_DRAGGING
            || state == BottomSheetBehavior.STATE_SETTLING) {
        mBottomSheetBehavior.setState(BottomSheetBehavior.STATE_COLLAPSED);
    } else {
        finishActivityIfNotFinishing();
    }
}
```
</action>

<task id="07-03-02">
<title>Ensure terminal process continuity by validating TermuxService binding is unaffected by sheet state</title>
<read_first>
- app/src/main/java/com/termux/app/TermuxActivity.java (onServiceConnected, service binding lifecycle)
- app/src/main/java/com/termux/app/TermuxService.java (service lifecycle — scan for onDestroy, onBind)
- .planning/phases/07-bottom-sheet-layout/07-RESEARCH.md (§R3 — TerminalView updateSize)
</read_first>
<action>
Terminal process continuity (LAYT-06) requires that sheet state changes never interrupt running processes. This is architecturally guaranteed because:
- Terminal processes are managed by `TermuxService` (bound service, independent of view lifecycle)
- `TermuxSession` objects live in the service, not the activity
- Sheet state changes only affect the view hierarchy, not the service binding

However, there is one concern from the research: rapid `TerminalView.onSizeChanged()` calls during sheet animation cause frequent PTY resize events. Add a debounce mechanism:

1. In `TermuxActivity.java`, in the `BottomSheetCallback.onStateChanged()` method (added in 07-01-02), add logic to temporarily suppress terminal resize during sheet animation:

Update the BottomSheetCallback in `setupBottomSheet()`:
```java
mBottomSheetBehavior.addBottomSheetCallback(new BottomSheetBehavior.BottomSheetCallback() {
    @Override
    public void onStateChanged(@NonNull View bottomSheet, int newState) {
        if (newState == BottomSheetBehavior.STATE_EXPANDED && mTerminalView != null) {
            mTerminalView.requestFocus();
        }
        // Re-enable terminal size updates when sheet settles
        if (newState != BottomSheetBehavior.STATE_DRAGGING
                && newState != BottomSheetBehavior.STATE_SETTLING) {
            if (mTerminalView != null) {
                mTerminalView.updateSize();
            }
        }
        Logger.logDebug(LOG_TAG, "Sheet state changed to: " + newState);
    }

    @Override
    public void onSlide(@NonNull View bottomSheet, float slideOffset) {
        // No-op — terminal resizes only on state settle, not during animation
    }
});
```

2. Verify that `TerminalView.updateSize()` is a public method. Check `TerminalView.java` — if `updateSize()` is not public, the callback should call a different method or the approach should be adjusted. If `updateSize()` is package-private or protected, just remove the explicit call — the `onSizeChanged()` override already handles resize when the view settles.

3. The key validation for LAYT-06 is that `TermuxService` binding in `onServiceConnected()` and `onServiceDisconnected()` is never triggered by sheet state changes. Verify there is no code path from sheet callbacks to service unbinding. This is a code review task — no code changes needed if the binding lifecycle is clean.
</action>

**5. Add startNestedScroll/stopNestedScroll calls in the GestureAndScaleRecognizer.Listener to prevent BottomSheetBehavior from intercepting terminal scrolls.**

The approach: TerminalView should ALWAYS consume its own scroll. The drag handle is the exclusive sheet drag zone. The NestedScrollingChild3 implementation signals to the parent "I'm scrolling, don't intercept me" by calling startNestedScroll, but TerminalView consumes everything itself.

> **Design note:** startNestedScroll is placed in `onScroll()` (guarded by `!scrolledWithFinger`) rather than in ACTION_DOWN because TerminalView delegates touch handling to GestureAndScaleRecognizer — there is no direct onTouchEvent ACTION_DOWN hook for scroll initiation. The `onScroll()` callback is the earliest point where we know a scroll gesture is happening.

In the `onUp()` callback (around line 143):
After `mScrollRemainder = 0.0f;` add:
```java
stopNestedScroll(ViewCompat.TYPE_TOUCH);
```

In the `onScroll()` method, before the existing scroll logic, add:
```java
if (!scrolledWithFinger) {
    startNestedScroll(ViewCompat.SCROLL_AXIS_VERTICAL, ViewCompat.TYPE_TOUCH);
}
```

Keep the existing scroll logic unchanged — TerminalView always consumes its own scroll. The `startNestedScroll` call tells BottomSheetBehavior that a nested scroll is happening, which prevents it from intercepting the touch.

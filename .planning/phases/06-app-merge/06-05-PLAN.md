<action>
Edit `.github/workflows/debug_build.yml` to add the build_runner step. Insert a new step between "Flutter pub get" and "Debug Flutter module setup":

```yaml
      - name: Flutter codegen (build_runner)
        run: |
          cd flutter_module
          dart run build_runner build --delete-conflicting-outputs
```

This generates:
- `*.freezed.dart` (Freezed models)
- `*.g.dart` (JSON serializable, Riverpod codegen)
- `soul_database.g.dart` (Drift)
- `objectbox.g.dart` (ObjectBox — will overwrite the committed version, which is fine)

The `--delete-conflicting-outputs` flag prevents build_runner from failing on stale generated files.

Also, the ObjectBox generator needs the `objectbox-model.json` to be present. Since it's committed in `flutter_module/lib/objectbox-model.json` (copied in Plan 01), this works out of the box.

Note: `envied_generator` would need a `.env` file. Since the SOUL app has no `@Envied` annotations (confirmed: grep returned zero results), this is not an issue. If VOYAGE_API_KEY is used, it's via `String.fromEnvironment()` which is resolved at compile time via `--dart-define`, not at codegen time.
</action>

<task id="06-05-02">
<title>Add dart-define flags for secrets in CI builds</title>

<read_first>
- /data/data/com.termux/files/home/soul-terminal/.github/workflows/debug_build.yml
- /data/data/com.termux/files/home/soul-terminal/app/build.gradle
</read_first>

<action>
The SOUL app uses `String.fromEnvironment('VOYAGE_API_KEY')` in EmbeddingService and `SENTRY_DSN` in SentryConfig. These are passed via `--dart-define` at build time. In add-to-app (flutter_module), dart-defines are passed via Gradle properties.

Add to the "Build APKs" step's environment or as Gradle properties. In `app/build.gradle`, the flutter module is built as part of the Gradle build. Dart-defines for add-to-app modules are passed via:

```groovy
// In the flutter module's build, dart-defines are set via the EXTRA_DART_DEFINES environment variable
// or via flutter.target property
```

The simplest approach: add environment variables to the "Build APKs" step:

```yaml
      - name: Build APKs
        shell: bash {0}
        env:
          VOYAGE_API_KEY: ${{ secrets.VOYAGE_API_KEY || '' }}
          SENTRY_DSN: ${{ secrets.SENTRY_DSN || '' }}
        run: |
          ...existing build script...
```

For these to reach the Flutter module via `String.fromEnvironment()`, they need to be passed as dart-defines through the Gradle build. In `flutter_module/.android/Flutter/build.gradle` (auto-generated), the `EXTRA_DART_DEFINES` property is read. Set it in the build step:

In the "Build APKs" run script, before `./gradlew assembleDebug`, add:
```bash
# Pass dart-defines to Flutter module via Gradle
export DART_DEFINES=""
if [ -n "$VOYAGE_API_KEY" ]; then
  DART_DEFINES="VOYAGE_API_KEY=$VOYAGE_API_KEY"
fi
if [ -n "$SENTRY_DSN" ]; then
  if [ -n "$DART_DEFINES" ]; then DART_DEFINES="$DART_DEFINES,SENTRY_DSN=$SENTRY_DSN"; else DART_DEFINES="SENTRY_DSN=$SENTRY_DSN"; fi
fi
if [ -n "$DART_DEFINES" ]; then
  export EXTRA_DART_DEFINES=$(echo "$DART_DEFINES" | tr ',' '\n' | while read d; do echo -n "$(echo -n "$d" | base64 -w0),"; done | sed 's/,$//')
fi
```

If secrets are not configured in the repo → build still succeeds (empty string fallback). The app gracefully handles missing keys (Sentry is a no-op, embeddings return null).
</action>

- `grep "EXTRA_DART_DEFINES\|dart-define" .github/workflows/debug_build.yml` returns a match

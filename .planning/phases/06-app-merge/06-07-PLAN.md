<tasks>

<task id="07.1">
<title>Update debug_build.yml met build_runner en dart-defines</title>
<read_first>
- .github/workflows/debug_build.yml
- flutter_module/android/build.gradle — check hoe dart-defines worden doorgegeven aan Flutter module builds
</read_first>
<action>
Voeg de volgende stap toe NA "Flutter pub get" en VOOR "Build APKs":

```yaml
- name: Flutter codegen (build_runner)
  run: |
    cd flutter_module
    dart run build_runner build --delete-conflicting-outputs
```

Voeg dart-defines toe aan de Gradle build stap. In een add-to-app module worden dart-defines doorgegeven via de `EXTRA_DART_DEFINES` Gradle property (base64-encoded, comma-separated). Controleer `flutter_module/android/build.gradle` voor de exacte property naam — standaard Flutter module builds accepteren dit via:

```groovy
// In flutter_module/android/build.gradle:
// project.property('extra-dart-defines') wordt automatisch verwerkt door Flutter's Gradle plugin
```

In de "Build APKs" stap, voeg de dart-defines toe als Gradle property:

```yaml
env:
  EXTRA_DART_DEFINES: "VOYAGE_API_KEY=${{ secrets.VOYAGE_API_KEY }},SENTRY_DSN=${{ secrets.SENTRY_DSN }}"
```

Als `String.fromEnvironment` gracefully faalt bij lege waarde (wat het geval is voor VOYAGE_API_KEY), dan is de dart-define optioneel en kan de build ook zonder secrets slagen.
</action>
<acceptance_criteria>
- `grep "build_runner" .github/workflows/debug_build.yml` vindt een match
- build_runner stap staat NA "Flutter pub get" en VOOR "Build APKs"
- `grep -E "VOYAGE_API_KEY|SENTRY_DSN|EXTRA_DART_DEFINES" .github/workflows/debug_build.yml` vindt matches
</acceptance_criteria>
</task>
